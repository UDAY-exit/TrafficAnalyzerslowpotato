What Is This Application?

Traffic Analyzer is an Android application that captures, parses, analyzes, and filters every network packet that passes through the device — without requiring root access. It is designed as a cybersecurity / SOC (Security Operations Center) classroom demo tool to visually show students what real network packets look like, how traffic can be categorized and filtered, and how IP reputation checks work in real time.

🎯
Learning Objective Students can see every packet leaving and entering their phone — source IP, destination IP, protocol, ports, TCP SYN flags — and apply live filter rules just like a professional firewall or IDS.

How VPN Packet Capture Works

The TUN Interface — Key Concept

Android's VpnService API lets any app (with user permission) create a virtual TUN network interface — essentially a fake network adapter that the OS treats as real. Once created, the OS routes all IPv4 traffic from the device through this interface. The app can read those raw bytes from a file descriptor.

📡
TUN vs TAP TUN operates at the network layer (Layer 3) — it sees raw IP packets. TAP operates at Layer 2 (Ethernet frames). Android VpnService uses TUN, so we get complete IPv4 headers but not Ethernet headers.

Step-by-Step Packet Flow

1. App on device makes any network request (browser, WhatsApp, YouTube…) ↓ 2. OS routes the packet to the TUN interface (virtual adapter we created) ↓ 3. TrafficVpnService reads raw bytes from ParcelFileDescriptor in a loop ↓ 4. IPv4Parser.parse(buffer) extracts header fields (IPs, ports, protocol, flags) ↓ 5. Packet object emitted on SharedFlow<Packet> ↓ 6. PacketViewModel collects each Packet via collectJob ↓ 7. PacketFilterEngine.apply(packet, config) returns PASS or DROP ↓ 8. filteredPackets StateFlow updated → Compose UI recomposes → row appears on screen

Critical Detail — Self-Exclusion

The VPN also captures the app's own traffic. Without exclusion, when the app calls the IPinfo API to look up a reputation, that very API call gets captured by itself — creating a loop and causing UnknownHostException: Unable to resolve host ipinfo.io. 

// In TrafficVpnService.establishVpn()
builder.addDisallowedApplication(packageName)
// This tells Android: route everything through TUN EXCEPT this app's own packets.

IPv4 Packet Parsing (RFC 791)

The IPv4Parser reads a raw byte array and extracts all fields we need from the standard IPv4 header format. No library is used — it is pure byte math.

Header Layout (what each byte means)

Byte  0  : Version (top 4 bits=4) + IHL — header length in 32-bit words
Bytes 2-3: Total Length of the packet
Byte  9  : Protocol  (6 = TCP,  17 = UDP,  1 = ICMP)
Bytes 12-15: Source IP address  (4 bytes → "192.168.1.5")
Bytes 16-19: Destination IP address

— TCP/UDP header starts immediately after the IP header (at offset IHL×4) —
Bytes IHL+0 to IHL+1 : Source Port
Bytes IHL+2 to IHL+3 : Destination Port
Byte  IHL+13          : TCP Flags
                        Bit 1 (0x02) = SYN flag ← new connection attempt

What We Extract Per Packet

FieldMeaningUsed For
sourceIpSender IP addressDisplay, filter, reputation lookup
destIpReceiver IP addressDisplay, local IP filter (RFC-1918)
sourcePortSender port numberDisplay
destPortReceiver port numberWeb port filter, port badge label
protocolTCP / UDP / ICMPProtocol filter, badge color
lengthPacket size in bytesDisplay
isSynTCP SYN flag set?SYN badge, SYN-only filter mode

The Filter Engine — 5-Stage Pipeline

Every captured packet passes through a 5-stage filter chain. Each stage independently decides to PASS or DROP. The engine short-circuits — as soon as one stage returns DROP, the remaining stages are skipped.

Packet → [Stage 1] Protocol Check tcpOnly? → DROP if not TCP → [Stage 2] Direction Check outboundOnly? → DROP if source is local (RFC-1918) → [Stage 3] SYN Flag Check synOnly? → DROP if not a SYN packet → [Stage 4] Port Range Check webPortsOnly?→ DROP if not port 80/443/8080/8443 → [Stage 5] Local Dest Check dropLocal? → DROP if destination is RFC-1918 → [Optional] Reputation abuseScore → DROP if score > threshold → PASSED ✓

Filter Modes — Presets

ModeWhat It ShowsStages Active
RAWEverything, zero filteringNone
CONNECTIONSTCP traffic going outboundStage 1 + 2
WEB_ONLYHTTP/HTTPS/web traffic onlyStage 1 + 2 + 4
SUSPICIOUS_ONLYSYN packets to external IPsStage 1 + 2 + 3 + 5
CUSTOMUser toggles each rule manuallyAny combination

RFC-1918 Private Address Ranges

Used by Stage 2 (outbound detection) and Stage 5 (local destination drop):

10.0.0.0    – 10.255.255.255    (10/8)
172.16.0.0  – 172.31.255.255   (172.16/12)
192.168.0.0 – 192.168.255.255  (192.168/16)
127.0.0.0   – 127.255.255.255  (loopback)
169.254.0.0 – 169.254.255.255  (link-local)

FilterConfig — All Toggles

data class FilterConfig(
    val tcpOnly:              Boolean = false,  // Stage 1
    val outboundOnly:         Boolean = false,  // Stage 2
    val synOnly:              Boolean = false,  // Stage 3
    val webPortsOnly:         Boolean = false,  // Stage 4
    val dropLocalDestination: Boolean = false,  // Stage 5
    val suspiciousOnly:       Boolean = false,  // Shorthand preset
    val abuseScoreThreshold:  Int     = 25,     // Reputation cutoff (0–100)
    val showDropped:          Boolean = false   // Show DROPs dimmed in list
)

Per-Session collectJob — Preventing Stale Packets

A common bug was packets continuing to arrive after the user pressed Stop. This was caused by three independent issues, all fixed with a per-session collectJob pattern: 

private var collectJob: Job? = null

fun stopCapture(context: Context) {
    collectJob?.cancel()    // ← Kill coroutine FIRST (no more packets processed)
    collectJob = null
    sendActionStop(context) // ← Tell service to stop
    _isCapturing.value = false
}

fun launchVpnService(context: Context) {
    collectJob?.cancel()    // Cancel any stale job from prior session
    clearPackets()
    startService(context)   // Service calls resetPacketFlow() — fresh SharedFlow
    collectJob = viewModelScope.launch {
        delay(300)          // Wait for TUN interface to establish
        packetFlow.collect { packet ->
            if (!_isCapturing.value) return@collect  // Safety guard
            // … run filter engine, update flows
        }
    }
}

IP Reputation — IPinfo API Integration

How It Works

When a user taps a packet row and presses ANALYZE, the app navigates to AnalysisScreen. The AnalysisViewModel calls the IPinfo REST API to fetch geographic and reputation data for the destination IP.

DetailValue
Base URLhttps://ipinfo.io/
EndpointGET /{ip}?token={your_token}
AuthenticationQuery parameter token (NOT Bearer header)
Key storagelocal.properties → BuildConfig.IPINFO_API_KEY
ReturnsCity, region, country, org (ISP), abuse score (0–100)

Adding Your API Key

# In local.properties (never commit this file to git)
IPINFO_API_KEY=your_token_here
IPINFO_BASE_URL=https://ipinfo.io/
// In app/build.gradle.kts
buildConfigField("String", "IPINFO_API_KEY", "\"${localProperties["IPINFO_API_KEY"]}\"")
buildConfigField("String", "IPINFO_BASE_URL", "\"${localProperties["IPINFO_BASE_URL"]}\"")
⚠️
Self-Exclusion Required Without addDisallowedApplication(packageName), the VPN intercepts the app's own API call to ipinfo.io, causing Unable to resolve host ipinfo.io. This was the most subtle bug in the entire project.

All Bugs Fixed & Lessons Learned

#BugRoot CauseFix
1 buildConfigField syntax error Value string had no escaped quotes Correct Kotlin string escaping in build.gradle.kts
2 Unable to resolve host ipinfo.io VPN TUN captured app's own traffic addDisallowedApplication(packageName)
3 Wrong API auth format Used Authorization: Bearer header Switched to @Query("token")
4 Token never passed to API ViewModel built header but didn't pass it Fixed ViewModel → Repository → ApiService chain
5 Black screen on startup (12s) Wrong windowBackground + slow Compose JIT SplashScreen API + windowBackground = #0A0F1E
6 Packets after Stop (service restarts) START_STICKY caused OS to restart service Changed to START_NOT_STICKY
7 Packets after Stop (old SharedFlow) Static SharedFlow lived across sessions resetPacketFlow() — fresh instance per session
8 Packets after Stop (coroutine) init {} block collect ran forever Per-session collectJob cancelled in stopCapture()
9 HorizontalDivider not found Used deprecated Divider from Material2 Replaced with HorizontalDivider (Material3)
10 themes.xml XML corruption PowerShell Set-Content stripped XML quotes Rewrote file using Write tool directly

Key Concepts Quick Reference

ConceptWhat It Means
TUN interfaceVirtual Layer-3 network adapter created by VpnService. OS routes IP packets through it.
RFC 791The IPv4 standard — defines the exact byte layout of every IPv4 packet header.
RFC 1918Reserved private IP ranges: 10.x.x.x, 172.16–31.x.x, 192.168.x.x — not routable on public internet.
TCP SYN flagThe first packet in a TCP connection. SYN = new connection attempt. Useful to detect scanning.
SharedFlowKotlin hot flow — emits to all active collectors. Used for the packet stream from VPN service.
StateFlowKotlin state holder — always has a current value. UI reads these to know what to display.
START_NOT_STICKYIf OS kills the foreground service, do NOT restart it automatically. Prevents ghost packet emission.
MVVMModel-View-ViewModel: UI observes ViewModel, ViewModel talks to Repository/Service, never reversed.
Short-circuit evalFilter stops at first DROP — no need to check remaining stages, saves CPU on every packet.
Abuse score0–100 value from IPinfo: how often this IP has been reported for malicious activity.
